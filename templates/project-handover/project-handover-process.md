# Процесс приёмки проекта в архитектурное курирование

[EN](project-handover-process.en.md) | **RU**

## Роли

| Роль                              | Ответственность                                                 |
|-----------------------------------|-----------------------------------------------------------------|
| **Архитектор**                    | Курирование архитектуры, аудит, рекомендации, контроль качества |
| **Руководитель проекта (PM)**     | Бизнес-контекст, сроки, ресурсы, stakeholders                   |
| **Технический лидер (Tech Lead)** | Техническая реализация, команда, код, инфраструктура            |

---

## Алгоритм приёмки

```
[Инициация] -> [Сбор информации] -> [Аудит] -> [Решение] -> [Онбординг]
    1              2-3                 4           5            6
```

---

## Фаза 1: Инициация (1 день)

### Входной триггер
- Запрос от PM/руководства на передачу проекта
- Формальное назначение архитектора куратором

### Действия

1. **Получить вводную от PM** (30-60 мин)
   - Бизнес-контекст и цели проекта
   - Текущее состояние и проблемы
   - Ожидания от архитектурного курирования
   - Ключевые stakeholders
   - Сроки и ограничения

2. **Запросить доступы**
   - Репозитории (GitLab)
   - Документация (Wiki/Confluence/Notion)
   - Мониторинг (Grafana, logs)
   - Task tracker (Tracker/Jira/YouTrack)
   - Коммуникации (Slack/Teams каналы)

3. **Назначить kickoff встречу** с PM + Tech Lead

### Артефакты
- [ ] Заполнена карточка проекта (название, PM, Tech Lead, контакты)
- [ ] Получены доступы
- [ ] Назначена kickoff встреча

---

## Фаза 2: Kickoff встреча (1-2 часа)

### Участники
- Архитектор
- Руководитель проекта (PM)
- Технический лидер (Tech Lead)

### Повестка

**Блок 1: Бизнес-контекст (PM, 20 мин)**
- Что делает система? Для кого?
- Ключевые бизнес-метрики (KPI)
- Планы развития на ближайшие 3-6 месяцев
- Основные боли и проблемы

**Блок 2: Техническая картина (Tech Lead, 30 мин)**
- Архитектура: из чего состоит система?
- Основные технологии и паттерны
- Известные проблемы и технический долг
- Команда: кто что делает?

**Блок 3: Ожидания и формат работы (все, 20 мин)**
- Что ожидают от архитектора?
- Какой уровень вовлечения нужен?
- Как будем взаимодействовать? (регулярные встречи, ad-hoc, async)
- Кто принимает финальные решения по архитектуре?

**Блок 4: Договорённости (10 мин)**
- Сроки аудита
- Формат отчёта
- Дата следующей встречи

### Вопросы для kickoff

**PM:**
- Какие бизнес-цели у проекта на этот квартал/год?
- Какие проблемы мешают достигать целей?
- Есть ли критичные сроки/релизы?
- Кто основные пользователи системы?

**Tech Lead:**
- Можешь показать диаграмму архитектуры? (или нарисовать на доске)
- Какие сервисы самые критичные?
- Где сейчас больше всего боли?
- Что бы ты изменил, если бы было время?
- Как устроен процесс разработки? (code review, CI/CD, тесты)

### Артефакты
- [ ] Записаны ключевые тезисы встречи
- [ ] Получена диаграмма архитектуры (или договорились кто сделает)
- [ ] Определены сроки аудита
- [ ] Согласован формат взаимодействия

---

## Фаза 3: Сбор информации (3-5 дней)

### Источники информации

| Источник     | Что смотрим                 | Кто даёт           |
|--------------|-----------------------------|--------------------|
| Документация | Архитектура, ADR, API specs | Tech Lead          |
| Код          | Структура, качество, тесты  | Tech Lead          |
| Мониторинг   | Метрики, алерты, инциденты  | Tech Lead / DevOps |
| Task tracker | Backlog, техдолг, баги      | PM / Tech Lead     |
| Stakeholders | Боли, ожидания              | PM                 |

### Чек-лист сбора

**Документация:**
- [ ] Архитектурная документация (C4, diagrams)
- [ ] ADR (Architecture Decision Records)
- [ ] API документация
- [ ] Runbooks и инструкции
- [ ] Onboarding guide для разработчиков

**Код и инфраструктура:**
- [ ] Список репозиториев
- [ ] CI/CD pipelines
- [ ] Тестовое покрытие
- [ ] Результаты линтеров

**Операционная информация:**
- [ ] Дашборды мониторинга
- [ ] История инцидентов (за 3-6 месяцев)
- [ ] SLA/SLO (если есть)
- [ ] Нагрузка и метрики производительности

### Async-вопросы Tech Lead'у

```
Привет! Готовлюсь к аудиту, нужна информация:

1. Где лежит архитектурная документация?
2. Есть ли ADR? Где?
3. Можешь скинуть ссылку на основной дашборд мониторинга?
4. Какой сервис/компонент сейчас самый проблемный?
5. Были ли критичные инциденты за последние 3 месяца?

Если чего-то нет - так и напиши, это тоже полезная информация.
```

## Фаза 4: Аудит (3-7 дней)

### Структура аудита

**День 1-2: Обзор архитектуры**
- Изучить диаграммы и документацию
- Понять структуру системы (сервисы, БД, очереди)
- Нарисовать свою версию диаграммы если её нет
- Выявить bounded contexts

**День 3-4: Глубокое погружение**
- Посмотреть код ключевых сервисов
- Проверить тестовое покрытие
- Посмотреть CI/CD
- Изучить мониторинг и алерты

**День 5-7: Анализ и отчёт**
- Сформулировать проблемы
- Приоритизировать рекомендации
- Заполнить шаблон system-audit.md
- Подготовить презентацию результатов

### Что искать

**Структурные проблемы:**
- Distributed monolith (тесно связанные сервисы)
- Shared database
- Циклические зависимости
- God services (делает слишком много)

**Операционные проблемы:**
- Отсутствие мониторинга
- Нет алертов или alert fatigue
- Отсутствие runbooks
- Нет DR плана

**Проблемы качества:**
- Низкое тестовое покрытие
- Отсутствие code review
- Нет CI/CD
- Устаревшие зависимости

## Фаза 5: Решение о приёмке (1 день)

### Встреча по результатам аудита

**Участники:** Архитектор, PM, Tech Lead

**Повестка (1 час):**
1. Презентация результатов аудита (20 мин)
2. Обсуждение критичных проблем (20 мин)
3. Согласование плана улучшений (15 мин)
4. Решение о приёмке (5 мин)

### Варианты решения

| Решение                 | Когда                         | Что дальше                            |
|-------------------------|-------------------------------|---------------------------------------|
| **Принят**              | Нет критичных блокеров        | Переход к онбордингу                  |
| **Принят с оговорками** | Есть проблемы, но управляемые | Фиксируем условия и сроки устранения  |
| **Не принят**           | Критичные блокеры             | Возврат на доработку, повторный аудит |

### Критерии приёмки (минимум)

- [ ] Есть понимание архитектуры (диаграмма)
- [ ] Есть доступ к коду и мониторингу
- [ ] Есть Tech Lead или ответственный за техническую часть
- [ ] Нет критичных security issues
- [ ] Система в продакшене и работает

### Артефакты
- [ ] Отчёт аудита (system-audit.md)
- [ ] Презентация результатов
- [ ] Согласованный план улучшений
- [ ] Решение о приёмке (зафиксировано)

---

## Фаза 6: Онбординг в курирование (ongoing)

### Первая неделя после приёмки

1. **Добавиться в коммуникации**
   - Slack/Teams каналы проекта
   - Рассылки по инцидентам
   - Приглашения на ключевые встречи

2. **Настроить регулярные touchpoints**
   - С Tech Lead: еженедельно 30 мин (техническое)
   - С PM: раз в 2 недели 30 мин (бизнес + приоритеты)

3. **Определить точки контроля**
   - Code review архитектурно-значимых изменений
   - Участие в design review новых фич
   - Ревью ADR

### Формат регулярного взаимодействия

**Еженедельный sync с Tech Lead (30 мин):**
- Что происходит на этой неделе?
- Какие технические решения принимаются?
- Нужна ли помощь/ревью?
- Прогресс по плану улучшений

**Bi-weekly с PM (30 мин):**
- Бизнес-приоритеты
- Новые требования и их влияние на архитектуру
- Ресурсы и сроки
- Эскалации (если есть)

### Точки эскалации

| Ситуация                                     | Действие                               |
|----------------------------------------------|----------------------------------------|
| Tech Lead не согласен с рекомендацией        | Эскалация на PM, совместное обсуждение |
| Критичный техдолг игнорируется               | Эскалация на PM + руководство          |
| Архитектурное решение противоречит стратегии | Фиксация в ADR, эскалация если надо    |

## Шаблоны коммуникаций

### Запрос информации (async)

```
Привет, [Tech Lead]!

Принял проект [название] в архитектурное курирование.
Готовлюсь к аудиту, нужна твоя помощь.

Можешь поделиться:
1. [Конкретный вопрос 1]
2. [Конкретный вопрос 2]
3. [Конкретный вопрос 3]

Если чего-то нет - скажи, это тоже важно знать.

Срок: до [дата], если возможно.
Спасибо!
```

### Приглашение на kickoff

```
Тема: Kickoff: приёмка проекта [название]

Привет!

Назначен архитектором-куратором проекта [название].
Предлагаю встретиться для знакомства и обсуждения.

Повестка:
1. Бизнес-контекст (PM)
2. Техническая картина (Tech Lead)
3. Ожидания и формат работы

Участники: [PM], [Tech Lead], [Архитектор]
Время: [дата, время]
Длительность: 1.5 часа

Подтвердите участие.
```

### Отправка результатов аудита

```
Тема: Результаты аудита проекта [название]

Привет!

Завершил аудит проекта [название].

Ключевые выводы:
- Общая оценка: [X]/5
- Критичных проблем: [X]
- Рекомендаций: [X]

Топ-3 проблемы:
1. [Проблема 1]
2. [Проблема 2]
3. [Проблема 3]

Полный отчёт: [ссылка]

Предлагаю встретиться [дата] для обсуждения.
```

## Метрики успеха курирования

| Метрика                     | Как измерять                    | Цель       |
|-----------------------------|---------------------------------|------------|
| Время приёмки               | Дни от запроса до решения       | < 2 недель |
| Закрытие критичных проблем  | % закрытых за квартал           | > 80%      |
| Удовлетворённость Tech Lead | Опрос раз в квартал             | > 4/5      |
| Участие в design review     | % архитектурных решений с ревью | > 90%      |
| Инциденты из-за архитектуры | Количество за квартал           | Снижение   |
