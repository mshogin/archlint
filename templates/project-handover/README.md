## Файлы

### [project-handover-process.md](project-handover-process.md)
**Процесс приёмки проекта в архитектурное курирование**

**Что внутри:**
- 6 фаз: Инициация → Kickoff → Сбор информации → Аудит → Решение → Онбординг
- Временные рамки: 7-14 дней
- Роли: Архитектор, PM, Tech Lead
- Шаблоны коммуникаций (email, встречи)
- Метрики успеха курирования

**Структура процесса:**

```
[Инициация] -> [Kickoff] -> [Сбор информации] -> [Аудит] -> [Решение] -> [Онбординг]
   1 день       1-2 часа        3-5 дней          3-7 дней    1 день      ongoing
```

**Когда использовать:**
- Принимаете проект в архитектурный надзор
- Нужен формальный процесс передачи проекта
- Хотите стандартизировать приёмку

**Ключевые фазы:**

**Фаза 1: Инициация (1 день)**
- Получить вводную от PM
- Запросить доступы (репозитории, мониторинг, документация)
- Назначить kickoff встречу

**Фаза 2: Kickoff встреча (1-2 часа)**
- Бизнес-контекст от PM
- Техническая картина от Tech Lead
- Согласовать ожидания и формат работы

**Фаза 3: Сбор информации (3-5 дней)**
- Документация (архитектура, ADR, API specs)
- Код и инфраструктура
- Операционная информация (мониторинг, инциденты, SLA)

**Фаза 4: Аудит (3-7 дней)**
- Обзор архитектуры (структура, bounded contexts)
- Глубокое погружение (код, тесты, CI/CD, мониторинг)
- Анализ и отчёт

**Фаза 5: Решение о приёмке (1 день)**
- Презентация результатов аудита
- Обсуждение критичных проблем
- Согласование плана улучшений
- Решение: Принят / С оговорками / Не принят

**Фаза 6: Онбординг (ongoing)**
- Добавиться в коммуникации
- Настроить регулярные touchpoints
- Определить точки контроля (code review, design review, ADR)

**Шаблоны коммуникаций:**
- Запрос информации (async)
- Приглашение на kickoff
- Отправка результатов аудита

**Метрики успеха:**
- Время приёмки: < 2 недель
- Закрытие критичных проблем: > 80% за квартал
- Удовлетворённость Tech Lead: > 4/5
- Участие в design review: > 90%

### [project-handover.md](project-handover.md)
**Чеклист для приёмки проекта**

**Что внутри:**
- 11 секций, 80+ пунктов проверки
- Критерии приёмки
- Оценки готовности (1-5)
- Три варианта решения

**Секции:**

1. **Бизнес-контекст**
   - Описание продукта
   - Бизнес-процессы (BPMN, интеграции, SLA)

2. **Системная аналитика**
   - Функциональные требования (backlog, user stories)
   - Нефункциональные требования (производительность, доступность, масштабируемость)
   - Модель данных (ER-диаграмма, объёмы)

3. **Архитектура**
   - Обзор (C4 Level 1-2, bounded contexts)
   - Компоненты системы
   - Интеграции
   - ADR (Architecture Decision Records)
   - Архитектурный аудит

4. **Код и репозитории**
   - Репозитории (доступ, CI/CD)
   - Качество кода (линтеры, code review, test coverage)
   - Зависимости

5. **Инфраструктура**
   - Среды (dev, staging, prod)
   - Infrastructure as Code (Terraform, Ansible)
   - CI/CD pipelines

6. **Операционная готовность**
   - Мониторинг и алертинг (Grafana, Prometheus, ELK)
   - Runbooks и инструкции
   - Бэкапы и DR (RTO, RPO)

7. **Безопасность**
   - Аутентификация и авторизация
   - Соответствие требованиям (security review, penetration test)

8. **Команда и знания**
   - Текущая команда
   - Knowledge transfer (сессии, FAQ, период поддержки)

9. **Открытые вопросы и риски**
   - Критические блокеры
   - Риски
   - Технический долг

10. **План действий после приёмки**
    - Ближайшие действия (1-2 недели)
    - Среднесрочные (1-3 месяца)
    - Долгосрочные улучшения

11. **Решение о приёмке**
    - Статус: Принят / С оговорками / Не принят
    - Условия приёмки (если с оговорками)
    - Подписи

**Когда использовать:**
- Во время фазы "Аудит" процесса приёмки
- Для структурированного сбора информации о проекте
- Чтобы зафиксировать состояние проекта

**Минимальные критерии приёмки:**
- [ ] Есть понимание архитектуры (диаграмма)
- [ ] Есть доступ к коду и мониторингу
- [ ] Есть Tech Lead или ответственный
- [ ] Нет критичных security issues
- [ ] Система в продакшене и работает

**Результат:**
- Документированное состояние проекта
- Оценки готовности по всем областям
- Список блокеров и рисков
- Формальное решение о приёмке

## Workflow приёмки

### Типичный сценарий (7-14 дней)

**День 1: Инициация**
```bash
# Получаем запрос
# Создаем карточку проекта
# Запрашиваем доступы
# Назначаем kickoff
```

**День 2: Kickoff**
```bash
# Встреча PM + Tech Lead (1-2 часа)
# Понять бизнес-контекст
# Понять техническую картину
# Согласовать ожидания
```

**День 3-7: Сбор информации + Аудит**
```bash
# Собираем документацию
# Изучаем код
# Смотрим мониторинг
# Заполняем project-handover.md
# Проводим system-audit (см. ../system-audit/)
```

**День 8: Решение**
```bash
# Презентуем результаты
# Обсуждаем проблемы
# Согласовываем план
# Принимаем решение: ПРИНЯТ / С ОГОВОРКАМИ / НЕ ПРИНЯТ
```

**День 9+: Онбординг**
```bash
# Настраиваем регулярные встречи
# Определяем точки контроля
# Начинаем работу
```

## Примеры использования

### Проект A: Distributed Monolith
**Аудит показал:**
- 5 сервисов тесно связаны
- Shared database
- Нет мониторинга алертов

**Решение:** Принят с оговорками
- План рефакторинга на 3 месяца
- Настроить алерты за 2 недели

### Проект B: Всё ОК
**Аудит показал:**
- Архитектура понятна
- Тесты, мониторинг, документация на месте
- Tech Lead компетентен

**Решение:** Принят
- Еженедельные sync с Tech Lead
- Участие в design review

### Проект C: Критичные проблемы
**Аудит показал:**
- Критичные security issues
- Tech Lead уходит через неделю
- Нет документации

**Решение:** Не принят
- Возврат на доработку
- Требования: устранить security issues, провести knowledge transfer
- Повторный аудит через месяц

## Best Practices

1. **Не пропускайте kickoff** - это самая важная встреча для понимания контекста
2. **Фиксируйте всё** - не полагайтесь на память, заполняйте чеклист
3. **Будьте объективны** - опирайтесь на метрики, не на впечатления
4. **Давайте план** - не только проблемы, но и конкретные действия
5. **Следите за сроками** - 2 недели максимум, иначе застрянете

## FAQ

**Q: Обязательно ли проходить все фазы?**

A: Да, для формальной приёмки. Можно ускорить, но не пропускать.

**Q: Что делать если Tech Lead недоступен?**

A: Эскалация на PM. Без Tech Lead полноценная приёмка невозможна.

**Q: Сколько проектов можно принимать параллельно?**

A: Рекомендуется 1-2 максимум. Каждый требует 50-70% времени на 2 недели.

**Q: Что если проект не принят?**

A: Фиксируете блокеры, возвращаете на доработку, назначаете повторный аудит.

**Q: Нужно ли заполнять все пункты чеклиста?**

A: Стремитесь к полноте. Минимум: бизнес-контекст, архитектура, операционная готовность, безопасность.
